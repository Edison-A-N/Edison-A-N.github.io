1. 背景与问题定义

核心挑战

Agent 的工具能力广泛，能解决的问题是广泛的。**当 Agent 解决领域问题涉及大量的工具的拼接组装，且拼接组装的 workflow 是由 LLM 理解和规划的**，我们需要有工具的管理能力和复杂的工具调用组织能力。

关键挑战：

  • 工具数量庞大：领域问题往往需要调用数十甚至上百个不同的工具
  • 工具组合复杂：工具之间的调用顺序、依赖关系、并行执行等需要动态规划
  • Workflow 动态生成：LLM 需要根据问题理解动态生成执行计划
  • 工具管理复杂：不同工具有不同的执行环境、依赖、版本等

设计目标

本架构旨在解决以上挑战，提供：

  • 灵活的工具编排：支持 LLM 动态生成和执行复杂的工具调用序列
  • 高效的资源管理：通过沙箱隔离和生命周期管理，优化资源使用
  • 可靠的执行环境：提供安全隔离的代码执行环境
  • 可扩展的服务架构：通过 MCP 协议实现功能代码的服务化

2. 核心架构概述

架构组件

  1. LLM 控制层

    • 生成控制代码（Control Code）
    • 负责动态组织 tool 的执行流程
    • 实现 LLM 单步可以执行一批 tool

  2. 代码执行沙箱

    • 轻量级执行环境
    • 支持特定网络访问能力
    • 隔离执行环境，确保安全性
    • 快速唤起和代码执行，降低延迟

  3. MCP 服务层

    • 将功能代码（Functional Code）服务化
    • 提供标准化的服务接口
    • 控制代码通过 MCP 协议调用功能代码服务
    • 功能代码一次编写多次执行，支持独立更新发布
    • 功能代码有复杂的执行环境需求，不适合在轻量级沙箱中部署和执行

  4. 数据存储层

    • 提供数据持久化能力
    • 支持数据的读写访问
    • 为沙箱和 MCP 服务提供数据支持

架构说明

关键特性

  • 职责分离：控制逻辑与功能实现分离
  • 安全隔离：沙箱环境限制执行权限
  • 网络访问控制：支持特定网络访问需求
  • 服务化架构：功能代码通过 MCP 协议提供服务

设计原则

控制代码与功能代码的职责边界

这是本架构的核心设计原则，决定了 LLM 生成的代码能做什么、不能做什么。

控制代码的职责（由 LLM 生成）：

  • 动态组织 tool 的执行流程
  • 实现 LLM 单步可以执行一批 tool
  • 负责编排和决策逻辑（if-else、循环、参数提取）
  • 仅允许：调用预定义的 MCP 工具、基本数据处理、流程控制

功能代码的职责（预定义的 MCP 工具）：

  • 提供经过安全审计的功能实现
  • 处理复杂的执行环境需求（GPU、数据库、外部 API）
  • 封装敏感操作（数据库访问、文件操作、网络请求）

为什么功能代码必须服务化：

  • 安全性：防止 LLM 生成的代码直接访问敏感资源
  • 一次编写多次执行：功能代码可以复用，避免重复编写
  • 独立更新发布：功能代码可以独立于控制代码进行更新和发布
  • 复杂执行环境需求：功能代码有自己复杂的执行环境需求，不适合在轻量级沙箱中部署
  • 沙箱定位：沙箱更适合轻量级环境以支持快速唤起和代码执行，降低延迟

3. 架构设计说明

3.1 控制平面（Control Plane）

3.1.1 动态编排器（Dynamic Orchestrator）

采用 Loop-based Re-Act 模式实现任务编排，维护一个 Think -> Generate Code -> Execute -> Observe 的状态机循环：

  1. Think：LLM 分析用户意图和当前上下文，规划执行步骤
  2. Generate Code：LLM 生成控制代码（Python），动态组织工具调用序列
  3. Execute：沙箱执行控制代码，调用 MCP 服务
  4. Observe：收集执行结果和错误信息，反馈给 LLM

状态管理：

  • 会话上下文：使用 Redis 存储会话的临时状态（Ephemeral State），确保多轮对话中上下文的一致性
  • 执行历史：记录每次代码执行的结果，支持 LLM 基于历史信息优化后续决策
  • Token Window 管理：控制平面负责压缩和传递上下文，避免超出 LLM 的 Token 限制
    • 语义摘要：对历史执行结果进行语义压缩，保留关键信息，减少 Token 占用
    • 向量记忆：将重要信息向量化存储，通过检索注入到上下文，避免直接占用 Token 窗口
    • 工具描述优化：当 MCP 服务数量庞大时，通过工具检索（Tool Retrieval）或技能筛选（Skills）机制动态筛选相关工具，仅将当前任务需要的工具描述注入到上下文，避免工具描述占满 Token 窗口

3.1.2 沙箱调度器（Sandbox Scheduler）

分配策略：

  • 分层预热池设计：

    • L1 池（内存级挂起）：维护已启动但挂起的沙箱实例，用于毫秒级响应（< 200ms）
    • L2 池（已启动容器）：维护已启动的容器实例，用于秒级扩容（< 2s）
    • L3 池（按需创建）：当预热池耗尽时，动态创建新实例

  • 亲和性调度（Affinity）：

    • 对于连续的多轮对话，尝试路由到同一沙箱实例
    • 复用本地缓存（如 pip install 的包），降低延迟
    • 平衡负载：当同一沙箱负载过高时，自动分配到新实例

生命周期管理：

  • 会话超时：每个沙箱会话设置超时时间（如 30 分钟），超时后自动销毁
  • 空闲回收：沙箱空闲一段时间（如 5 分钟）后，回收到预热池而非销毁
  • 资源上限：设置全局沙箱数量上限，达到上限时优先回收空闲实例
  • 硬超时保护：设置严格的 hard_timeout（如 30s），防止死循环代码占用资源

3.1.3 MCP 服务发现与可扩展性

服务注册与发现：

采用服务注册中心模式：

  • 服务注册：MCP 服务启动时向注册中心（如 Consul、etcd）注册服务信息（服务名、版本、端点、健康状态）
  • 服务发现：沙箱通过注册中心查询可用的 MCP 服务列表
  • 动态更新：服务上线/下线时，注册中心自动更新服务列表，沙箱实时感知

MCP Capability Aggregator（能力聚合器）：

  • 能力收集：当 Agent 启动时，聚合器向所有注册的 MCP Servers 发送 initialize 请求
  • 工具列表合并：收集所有 tools、resources、prompts 列表，合并后注入到 LLM 的 System Prompt
  • 动态更新：当新的 MCP Server 上线时，自动更新工具列表，无需重启 Agent
  • 版本管理：支持同一服务的多个版本，通过版本号区分工具能力
  • 工具筛选与过滤：当 MCP 服务数量庞大时（如数百个工具），采用工具检索机制（Tool Retrieval）或技能筛选（Skills）策略，根据当前上下文和任务需求动态筛选相关工具，避免所有工具描述占满 Token 窗口。工具筛选可基于语义相似度、历史使用模式、任务类型等维度进行

服务版本管理：

  • 语义化版本：MCP 服务使用语义化版本号（如 v1.2.3）
  • 多版本共存：支持同一服务的多个版本同时运行，通过版本号区分
  • 向后兼容：新版本必须保证向后兼容，不兼容变更需要升级主版本号

负载均衡与扩展：

  • 客户端负载均衡：沙箱端实现负载均衡（如轮询、最少连接），选择最优服务实例
  • 健康检查：定期检查服务健康状态，自动剔除不健康的实例
  • 水平扩展：MCP 服务支持水平扩展，新实例自动注册到服务发现系统

3.1.4 安全围栏（Security Fence）

请求级别拦截：控制平面在请求进入沙箱前进行安全策略检查：

  • 代码静态分析：检查控制代码是否包含危险操作（如文件系统访问、网络请求）
  • 工具调用白名单：验证代码中调用的 MCP 工具是否在授权列表中
  • 资源配额检查：验证请求是否超出资源配额限制

运行时监控：沙箱执行过程中，控制平面持续监控：

  • 资源使用监控：CPU、内存、网络使用情况
  • 异常行为检测：检测死循环、资源耗尽等异常情况
  • 自动终止：当检测到异常时，自动终止执行并回收资源

3.1.5 LLM 安全防护（LLM Security Protection）

Prompt 注入防护：

  • 输入过滤：对用户输入进行预处理，检测和过滤潜在的注入攻击模式
  • 上下文隔离：将用户输入与系统提示词隔离，防止用户输入污染系统指令
  • 输出验证：对 LLM 生成的代码进行静态分析，检测是否包含恶意逻辑
  • 多模型共识：关键决策点使用多个 LLM 模型生成代码，通过共识机制验证一致性

模型投毒检测：

  • 行为监控：监控 LLM 生成代码的异常模式，检测是否存在系统性绕过安全机制的行为
  • 输出分布分析：分析生成代码的统计特征，识别异常分布模式
  • 对抗样本检测：检测针对 LLM 的对抗攻击，如精心构造的输入导致生成恶意代码

提示词泄露防护：

  • 错误信息脱敏：沙箱执行错误时，对错误信息进行脱敏处理，避免泄露系统内部信息
  • 日志过滤：对日志中的敏感信息（如提示词片段、系统架构）进行过滤
  • 访问控制：限制调试信息的访问权限，仅授权人员可查看完整错误堆栈

LLM 输出验证层：

在 LLM 生成代码后、提交沙箱执行前，增加专门的验证层：

  • 静态代码分析：使用 AST（抽象语法树）分析，检测危险操作模式
  • 符号执行：对生成代码进行符号执行，验证基本正确性（如变量定义、函数调用）
  • 逻辑验证：检测循环依赖、未定义变量、类型错误等逻辑问题
  • 工具调用验证：验证代码中调用的 MCP 工具是否存在、参数类型是否正确

3.2 数据平面（Data Plane）

3.2.1 引用传递机制与数据流（Reference Passing Mechanism & Data Flow）

核心设计原则：控制流与数据流分离

这是工具编排架构的关键设计原则：LLM 只处理控制流（代码、逻辑、引用），数据平面处理实际的数据流（文件、大数据对象）。

引用传递机制（Reference Passing Mechanism）：

采用引用传递 + 共享存储模式，实现控制流与数据流的解耦：

  • 小数据（< 1MB）：直接通过 MCP 协议参数传递（JSON），LLM 生成的代码可以直接处理
  • 大数据（≥ 1MB）：采用引用传递模式
    • 大数据对象存储到对象存储（S3/MinIO），生成唯一引用 ID（如 artifact://id-123）
    • LLM 生成的代码只传递引用 ID，而不是实际数据
    • 后续工具通过引用 ID 从对象存储中获取实际数据
    • 优势：避免在控制代码中传输大文件，降低 Token 消耗和网络开销

引用管理（Reference Management）：

数据平面维护 ID -> S3 Path 的映射关系：

  • 引用生成：工具 A 运行后返回 {"status": "success", "data_ref": "artifact://id-123"}
  • 引用传递：LLM 生成的代码将这个引用传递给工具 B：mcp_call("tool_b", {"input_ref": "artifact://id-123"})
  • 引用解析：工具 B 通过引用 ID 从对象存储中获取实际数据
  • 生命周期管理：引用关联的数据在 workflow 完成后自动清理，或根据保留策略保留

控制代码的状态管理：

采用沙箱内存 + 外部持久化的混合模式：

  • 运行时状态：控制代码执行过程中的临时变量、中间结果存储在沙箱内存中，支持快速访问
  • 持久化状态：关键状态（如 workflow 进度、检查点）持久化到 Redis，支持会话恢复
  • 状态同步：沙箱销毁前自动同步关键状态到外部存储

运行时状态层（Execution State Layer）

解决”Workflow 动态生成与回溯”的问题，支持 LLM 自我修正（Self-Correction）和问题诊断：

  • 执行追踪存储（Execution Trace Store）：

    • 使用 TimescaleDB 或 Redis Streams 记录每一步的完整执行信息
    • 记录内容：Input（输入参数）、Code（生成的代码）、Output（输出结果）、Stderr（错误信息）、Resource Usage（资源使用情况）
    • 用途：对 Debug 和 LLM 自我修正（Self-Correction）至关重要，LLM 可以基于历史执行信息优化后续决策
    • 查询能力：支持按时间范围、工具名称、执行状态等维度查询执行历史

  • 变量存储（Variable Store）：

    • 使用 Redis 存储热数据，用于沙箱和外部 MCP 服务之间的轻量级数据交换
    • 存储内容：临时变量、中间计算结果、会话状态
    • 访问特点：高频读写、低延迟（< 10ms）

沙箱状态持久化：

  • 会话状态：沙箱会话期间的状态保存在沙箱内存中，支持多轮代码执行
  • 检查点机制：长时间运行的 workflow，定期创建检查点保存到 Redis
    • 检查点内容：代码执行位置、变量快照、引用映射关系
  • 会话恢复：沙箱异常退出后，可以从检查点恢复执行

数据一致性：

  • 单沙箱单线程：每个沙箱会话单线程执行控制代码，避免并发冲突
  • 工具调用原子性：单个工具调用保证原子性，失败时回滚
  • 最终一致性：跨工具的数据传递采用最终一致性模型，通过重试保证最终一致

分布式事务处理：

对于跨多个 MCP 服务的复杂 workflow，需要分布式事务保证数据一致性：

  • Saga 模式：采用 Saga 分布式事务模式，将长事务分解为多个本地事务
    • 编排式 Saga：由控制代码（编排器）协调各个 MCP 服务的本地事务
    • 补偿操作：每个 MCP 服务提供补偿操作（Compensating Action），用于撤销已执行的操作
    • 事务状态管理：在 Redis 中维护事务状态，支持事务恢复和回滚
  • 两阶段提交（2PC）：对于强一致性要求的场景，使用 2PC 协议
    • 准备阶段：协调器向所有参与者发送准备请求
    • 提交阶段：所有参与者准备成功后，协调器发送提交请求
    • 超时处理：设置超时机制，超时后自动回滚
  • 事务协调器：实现独立的事务协调器服务，管理分布式事务的生命周期
    • 事务注册：workflow 开始时注册事务，分配唯一事务 ID
    • 状态追踪：追踪每个参与服务的事务状态（未开始、进行中、已提交、已回滚）
    • 回滚协调：当部分服务失败时，协调器触发所有已成功服务的补偿操作

3.2.2 资产注册层与数据存储架构（Registry & Storage Architecture）

资产注册层（Registry & Discovery Layer）

解决”LLM 如何知道有哪些工具可用”的问题。当工具数量庞大（100+）时，无法将所有工具的详细 API 定义全部塞入 LLM Context Window，需要通过注册中心进行工具发现和检索。

工具注册中心（Tool Registry）：

采用 PostgreSQL + Vector Index（如 pgvector） 架构：

  • 工具元数据存储（PostgreSQL）：

    • 工具名称、版本、Docker 镜像地址、资源需求（CPU、内存、GPU）
    • 工具 Schema（OpenAPI/JSON Schema）：用于生成准确的调用代码
    • 工具依赖关系：工具间的 I/O 约束和硬性依赖（如工具 A 输出 DataFrame，工具 B 需要 CSV File）
    • 工具健康状态：服务可用性、版本信息

  • 工具语义索引（Vector Index）：

    • 存储工具的语义描述（Vector Embedding）：用于 RAG-based Tool Selection
    • 场景：当 LLM 面对用户请求”分析上个月的销售数据异常”时，通过语义检索找到 query_sales_db、anomaly_detection_algorithm、generate_report 等相关工具
    • 必要性：极高（针对 Tool Selection 阶段），解决工具数量庞大的问题

  • 依赖关系图（Dependency Graph）：

    • 不使用通用知识图谱：本架构不使用通用的知识图谱作为业务知识库
    • 轻量级依赖解析器：如果工具依赖关系复杂，在注册中心中实现轻量级的依赖解析器，基于元数据表运行
    • 依赖关系存储：使用关系型数据库的邻接表或图数据库（如 Neo4j，仅在依赖关系极其复杂时使用）

Prompt/Template Store：

  • 存储针对特定工具的高质量 Prompt 模板和 Few-shot examples
  • 用于提高 LLM 生成代码的准确性和一致性

数据存储架构（Storage Architecture）

采用热-温-冷三层存储架构，根据数据访问频率和重要性选择存储类型：

  • 热存储（Redis）：

    • 存储内容：会话状态、MCP 工具列表缓存、临时变量、检查点
    • 访问特点：高频读写、低延迟要求（< 10ms）
    • 生命周期：临时数据，TTL 自动过期

  • 温存储（S3/MinIO）：

    • 存储内容：用户上传的文件、工具生成的图表/报告、执行日志、中间结果、Artifacts（大文件、模型权重）
    • 访问特点：中频读写、可接受延迟（< 100ms）
    • 生命周期：根据业务需求设置保留策略

  • 冷存储（PostgreSQL）：

    • 存储内容：执行记录、审计日志、计费数据、元数据
    • 访问特点：低频读写、强一致性要求
    • 生命周期：长期保存，支持查询和分析

数据访问模式：

  • 统一存储接口：提供统一的存储抽象层，屏蔽底层存储差异
  • 沙箱访问：沙箱通过 SDK 访问存储，支持热存储和温存储
  • MCP 服务访问：MCP 服务通过统一接口访问所需存储，支持所有存储类型

数据隔离：

  • 租户隔离：通过命名空间或前缀实现多租户数据隔离
  • 任务隔离：每个任务有独立的数据目录，避免数据泄露
  • 访问控制：基于角色的访问控制（RBAC），确保数据安全

3.2.3 上下文与索引服务（Context & Index Service）

核心挑战：在工具编排场景中，存在两类上下文溢出问题：

  1. 规划阶段的上下文溢出：工具数量庞大（100+），无法将所有工具描述全部塞入 Context Window
  2. 执行阶段的上下文溢出：MCP Server 返回的大量数据（如 10MB 的 JSON）如何放入有限的 LLM Context Window？

上下文服务架构：

采用双向量存储架构，区分静态资源索引和运行时记忆：

1. 元数据向量存储（Meta Vector Store） - 解决规划阶段的上下文溢出

用于存储静态资源（Static Assets），解决工具发现和代码生成问题：

  • 工具索引（Tool Index）：

    • 存储工具的语义描述（Vector Embedding），用于 RAG-based Tool Selection
    • 场景：LLM 在生成代码前，需要先决定用什么工具。通过语义检索找到相关工具，而不是把所有工具都看一遍
    • 实现：与 3.2.2 的工具注册中心集成，使用相同的 Vector Index

  • Few-Shot Example 检索（Example Retrieval）：

    • 存储类似任务的成功代码片段（Vector Embedding）
    • 场景：为了让生成的代码更准确，检索”类似任务的成功代码片段”作为参考
    • 实现：将历史成功的 workflow 代码片段向量化存储，按需检索

2. 运行时向量存储（Runtime Vector Store） - 解决执行阶段的上下文溢出

用于存储执行过程数据（Episodic Memory），解决 Token Window 管理问题：

  • 执行历史记忆（Execution History Memory）：
    • 存储执行日志、中间结果摘要（Vector Embedding）
    • 场景：”刚才执行的步骤 1 输出了什么？”、”之前的对话历史是什么？”
    • 实现：将 MCP 服务返回的结果进行摘要和向量化，按需检索注入上下文

结果处理器（Result Processor）：

在 MCP 服务返回结果后、进入 LLM Context 之前，必须进行数据清洗和处理：

  • 自动截断：当结果超过 Token 限制时，自动截断并保留关键信息
    • 智能截断：优先保留结构化数据的关键字段，丢弃冗余信息
    • 摘要生成：对大数据集生成摘要，保留统计信息和关键样本
  • 数据摘要：对复杂数据结构进行语义摘要
    • 表格数据：保留表头和关键行，生成统计摘要
    • 列表数据：保留前 N 项和后 N 项，中间部分用摘要替代
    • 嵌套结构：递归处理嵌套数据，保留顶层结构和关键叶子节点
  • 向量化存储：将重要信息向量化存储到 Runtime Vector Store，通过检索注入到上下文
    • 语义检索：当 LLM 需要详细信息时，通过向量检索获取相关片段
    • 按需加载：避免一次性加载所有数据，按需检索和注入

Token Window 管理策略：

  • 分层压缩：
    • Level 1（原始数据）：完整保留，用于沙箱执行
    • Level 2（摘要数据）：压缩后传递给 LLM，用于决策
    • Level 3（向量索引）：向量化存储到 Runtime Vector Store，按需检索
  • 上下文窗口分配：
    • 用户 Prompt：保留完整用户输入
    • 工具描述：通过 Meta Vector Store 动态加载相关工具描述（而非全部工具）
    • 执行历史：保留最近 N 次执行结果，历史结果通过 Runtime Vector Store 检索
    • 当前结果：当前工具调用结果，经过结果处理器处理

数据传递策略：

  • 小数据（< 1MB）：直接通过 MCP 协议的 JSON-RPC 传输，使用 JSON 序列化
  • 大数据（≥ 1MB）：采用引用传递模式（见 3.2.1），大数据对象存储到对象存储（S3/MinIO），通过引用 ID 传递
  • 流式传输：支持流式传输大文件，避免内存溢出

3.3 基础设施平面（Infrastructure Plane）

3.3.1 MCP 网关与网络拓扑

核心挑战：沙箱通常是断网的或受限的，它如何通过安全的隧道连接到部署在内网或其他容器中的 MCP Server？

MCP 网关（MCP Gateway）：

MCP 网关是架构的核心枢纽，负责协议转换、鉴权与路由：

  • 协议转换：将沙箱内的函数调用（如 client.call_tool()）转换为 MCP 协议流
    • 支持多种传输方式：Stdio、SSE（Server-Sent Events）、HTTP
    • 协议适配：将沙箱内的 Python/JavaScript 调用转换为标准 MCP JSON-RPC 消息
  • 鉴权与路由：确保沙箱只能调用被授权的 MCP 服务
    • 工具调用白名单：验证代码中调用的 MCP 工具是否在授权列表中
    • 服务路由：根据工具名称和服务发现信息，路由到正确的 MCP Server 实例
    • 权限验证：验证当前会话是否有权限调用特定工具

网络拓扑：

沙箱 ↔ 代理 ↔ MCP Server 的连接方式：

  • Sidecar 模式：每个沙箱实例配备一个 Sidecar 代理，负责 MCP 协议转换和网络访问
    • 优势：沙箱完全隔离，所有外部通信必须通过 Sidecar
    • 实现：使用 Unix Socket 或本地 HTTP 连接沙箱与 Sidecar，Sidecar 通过安全通道连接 MCP Server
  • 集中式网关模式：所有沙箱通过统一的 MCP 网关访问 MCP 服务
    • 优势：集中管理、统一鉴权、易于监控
    • 实现：沙箱通过内部网络连接到网关，网关负责协议转换和路由
  • 传输方式选择：
    • Stdio：适用于本地部署，通过标准输入输出通信
    • SSE（Server-Sent Events）：适用于 Web 环境，支持流式响应
    • HTTP：适用于分布式部署，支持负载均衡和健康检查

安全隔离：

  • 网络命名空间隔离：使用 Linux Network Namespace 隔离沙箱网络
  • 最小权限原则：沙箱只能访问 MCP 网关，禁止直接访问外部网络
  • 白名单机制：基于白名单的访问控制，只允许访问预定义的 MCP 服务
  • 系统调用拦截：在沙箱层面拦截危险系统调用（如网络相关系统调用）

3.3.2 容器与虚拟化技术

技术选型对比：

技术方案 启动时间 隔离性 资源开销 适用场景
Firecracker ~125ms 硬件级隔离（KVM） 低（microVM） 生产环境，高安全要求
gVisor ~200ms 用户空间内核 中 开发环境，中等安全要求
Docker ~500ms 命名空间隔离 低 开发环境，快速迭代
WebAssembly ~50ms 沙箱隔离 极低 轻量级任务，浏览器环境

选型决策（ADR）：

选择 Firecracker 的原因：

  • 启动速度快：125ms 启动时间满足毫秒级响应要求
  • 硬件级隔离：基于 KVM，提供更强的安全隔离，防止沙箱逃逸
  • 资源效率：microVM 设计，资源开销低，支持高密度部署
  • 生产就绪：AWS Lambda 等生产环境验证，稳定可靠

不选择 Docker 的原因：

  • 隔离性不足：命名空间隔离较弱，存在容器逃逸风险
  • 启动时间较长：500ms 启动时间无法满足毫秒级响应要求

不选择 WebAssembly 的原因：

  • 功能限制：WASM 环境受限，无法支持完整的 Python/Node.js 运行时
  • 生态不成熟：工具链和调试工具不够完善

网络隔离策略：

  • Network Policies：使用 Kubernetes Network Policies 限制 Pod 间通信
  • 服务网格：Istio/Linkerd 实现细粒度的流量控制和 mTLS 加密
  • 防火墙规则：iptables/nftables 限制出站流量，仅允许访问 MCP 网关

防止沙箱逃逸：

  • 系统调用过滤：使用 seccomp 过滤危险系统调用
  • 文件系统只读：沙箱文件系统只读，临时文件写入到挂载的卷
  • 资源限制：cgroups 限制 CPU、内存、网络带宽
  • 审计日志：记录所有系统调用，支持安全审计

3.4 可靠性工程（Reliability & Operations）

本架构采用了基础设施层与认知层双重容错机制，以应对 MCP 工具调用失败及代码执行错误。

3.4.1 认知层自愈（Cognitive Self-Healing）

当沙箱执行返回 stderr 或异常退出码时，控制平面不会直接向用户报错，而是将错误堆栈（Stack Trace）作为新的 Observation 回传给 LLM，触发自愈循环：

  1. 错误捕获：沙箱执行失败时，捕获完整的错误信息（包括 stderr、异常类型、堆栈跟踪）
  2. 错误分析：将错误信息注入到 LLM 的上下文中，要求 LLM 分析错误原因
  3. 代码修复：LLM 基于错误分析，生成修正后的控制代码
  4. 重试执行：在沙箱中执行修复后的代码

重试策略：

  • 最大重试次数：设定 max_repair_attempts=3，避免无限循环
  • 错误分类：区分可修复错误（如参数错误、逻辑错误、类型错误）和不可修复错误（如权限错误、资源不足）
  • 渐进式修复：每次重试时，LLM 可以访问之前的错误历史，逐步改进修复方案
  • 错误上下文保留：保留完整的错误链，帮助 LLM 理解错误的根本原因

3.4.2 MCP 服务熔断与降级

断路器（Circuit Breaker）：

对每个 MCP Server 维护断路器状态机（Closed → Open → Half-Open）：

  • Closed（正常）：服务正常，请求正常通过
  • Open（熔断）：服务连续失败超过阈值（如 5 次），进入熔断状态，直接拒绝请求
  • Half-Open（半开）：熔断后等待恢复时间（如 30s），尝试放行少量请求测试服务是否恢复

降级策略：

当关键 MCP 服务不可用时，系统自动在 Prompt 中注入降级提示：

  • 工具不可用提示："当前搜索工具不可用，请尝试使用已知知识回答"
  • 替代方案引导：引导 LLM 使用其他可用工具或简化任务目标
  • 部分功能降级：允许 workflow 在部分工具失败时继续执行，跳过不可用的步骤

重试策略：

  • 指数退避重试：采用带抖动的指数退避策略处理网络抖动
    • 初始延迟：100ms
    • 最大延迟：5s
    • 抖动范围：±20%
  • 超时设置：为不同类型的工具设置不同的超时时间
    • 快速工具（如数据库查询）：5s
    • 计算密集型工具（如分子对接）：60s
    • 硬超时保护：所有工具调用设置硬超时（如 120s），防止无限等待

3.4.3 监控与可观测性（Monitoring & Observability）

分层监控体系：

采用分层监控策略，从基础设施到业务层全面覆盖：

L1 基础设施监控：

  • 沙箱指标：
    • 启动时间：沙箱从创建到就绪的时间（目标 < 200ms）
    • 资源使用：CPU、内存、网络使用情况
    • Warm Pool 深度：预热池中可用实例数量（告警阈值：< 5 instances）
    • 存活率：沙箱实例的存活率，检测异常退出
  • MCP 服务指标：
    • 服务健康状态：每个 MCP Server 的健康检查结果
    • 实例数量：每个服务的运行实例数
    • 资源使用：服务实例的 CPU、内存使用情况

L2 应用监控：

  • API 指标：
    • 请求速率（RPS）：每秒处理的请求数
    • 错误率：请求失败率（告警阈值：> 5%）
    • 延迟分布：P50、P95、P99 延迟
  • MCP 工具调用指标：
    • 调用延迟：每个工具的调用延迟（告警阈值：P99 > 5s）
    • 成功率：工具调用的成功率（告警阈值：< 90%）
    • 错误率：工具调用的错误率，按错误类型分类
    • 吞吐量：每秒工具调用次数

L3 认知监控（LLM Specific）：

  • Token 消耗：
    • Token 使用速率：每秒消耗的 Token 数
    • 每任务 Token 成本：单个任务消耗的 Token 数（告警阈值：> $0.50）
    • Token 窗口利用率：Context Window 的使用率
  • 任务执行指标：
    • 任务完成率：成功完成的任务占比（目标：> 90%）
    • 步骤数分布：每个任务的执行步骤数（告警阈值：> 15 steps，可能陷入死循环）
    • 平均修复次数：认知层自愈的平均重试次数
    • 幻觉检测：检测 LLM 生成代码中的逻辑错误和幻觉

全链路追踪（Distributed Tracing）：

采用分布式追踪系统（OpenTelemetry、Jaeger、Zipkin）实现全链路追踪：

  • Trace ID 贯穿：从用户 Prompt 到 LLM 生成 → 沙箱执行 → MCP 调用 → 结果返回，唯一的 trace_id 贯穿全流程
  • Span 记录：记录每个关键操作的 Span（LLM 调用、代码生成、沙箱执行、MCP 调用）
  • 上下文传播：Trace ID 通过 MCP 协议头传播到所有下游服务
  • 性能分析：通过 Trace 分析每个环节的耗时，识别性能瓶颈

错误追踪：

  • 错误聚合：按错误类型、服务、时间窗口聚合错误
  • 根因分析：通过 Trace 关联错误发生的完整链路
  • 告警机制：当错误率超过阈值时，自动触发告警

业务指标：

  • 任务完成率：成功完成的任务占比
  • 工具调用成功率：MCP 工具调用的成功率
  • 平均修复次数：认知层自愈的平均重试次数
  • 资源利用率：沙箱、MCP 服务的资源使用效率

监控指标矩阵：

监控维度 关键指标 告警阈值示例 业务价值
基础设施 Sandbox Warm Pool Depth < 5 instances 保证秒级启动体验
MCP 层 Tool Execution Latency (P99) > 5s 识别拖慢 Agent 的外部工具
MCP 层 Tool Error Rate > 10% 发现工具故障或参数格式变更
认知层 Step Count per Task > 15 steps 识别 Agent 陷入死循环
认知层 Token Usage / Cost per Task > $0.50 成本控制与异常流量检测
业务层 Task Success Rate < 90% 核心业务健康度

3.4.4 调试与问题诊断（Debugging & Diagnostics）

针对动态生成代码的调试工具和问题诊断机制。

代码可读性增强：

  • 代码格式化：对 LLM 生成的代码进行格式化，提高可读性
  • 变量重命名：自动重命名混乱的变量名，使用有意义的名称
  • 注释生成：为生成的代码自动生成注释，说明关键逻辑
  • 代码简化：检测并简化冗余逻辑，提高代码质量

调试工具：

  • 断点支持：支持在动态生成的代码中设置断点，暂停执行
  • 单步执行：支持单步执行代码，逐行查看执行过程
  • 变量查看：实时查看执行过程中的变量值和状态
  • 调用栈追踪：显示完整的函数调用栈，包括 MCP 服务调用链
  • 交互式调试：提供交互式调试界面，支持修改变量值、跳过代码段等操作

问题复现机制：

  • 执行快照：记录每次代码执行的完整快照，包括代码、输入、输出、环境状态
  • 确定性执行：通过记录随机种子、时间戳等方式，支持确定性复现
  • 版本关联：将执行快照与 LLM 模型版本、MCP 服务版本关联
  • 回放功能：支持回放历史执行，重现问题场景

日志与追踪：

  • 结构化日志：使用结构化日志格式，便于查询和分析
  • 日志分级：区分不同级别的日志（DEBUG、INFO、WARN、ERROR），支持按级别过滤
  • 上下文关联：日志中包含 Trace ID，关联到全链路追踪系统
  • 敏感信息过滤：自动过滤日志中的敏感信息（如密码、令牌）

性能分析：

  • 执行时间分析：分析每个代码段的执行时间，识别性能瓶颈
  • 资源使用分析：分析 CPU、内存、网络等资源使用情况
  • 调用链分析：分析 MCP 服务调用链的性能，识别慢调用
  • 优化建议：基于性能分析结果，提供代码优化建议

3.5 开发者体验与生态（Developer Experience & Ecosystem）

MCP 服务开发框架：

开发工具链：

  • SDK 支持：提供多语言 SDK（Python、JavaScript、Go、Rust），简化 MCP 服务开发
  • 代码生成：基于接口定义自动生成服务框架代码，减少样板代码
  • 本地测试环境：提供本地测试环境，支持开发者本地调试和测试
  • Mock 服务：提供 Mock 服务框架，支持单元测试和集成测试

服务模板：

  • 标准模板：提供标准化的服务模板，包含最佳实践和示例代码
  • 领域模板：针对不同领域（数据查询、计算、分析）提供专门的模板
  • 快速启动：通过模板快速创建新服务，加速开发流程

测试策略：

  • 单元测试：支持单元测试，验证服务功能的正确性
  • 集成测试：支持集成测试，验证服务与沙箱、存储的集成
  • 端到端测试：支持端到端测试，验证完整 workflow 的执行
  • 性能测试：提供性能测试工具，验证服务的性能指标
  • 兼容性测试：自动测试服务版本升级的兼容性

调试工具：

  • 本地调试：支持本地调试 MCP 服务，设置断点、查看变量
  • 远程调试：支持远程调试生产环境中的服务（通过安全通道）
  • 日志查看：提供统一的日志查看界面，支持过滤和搜索
  • 性能分析：提供性能分析工具，识别性能瓶颈

文档与契约：

接口文档：

  • OpenAPI/Swagger：使用 OpenAPI/Swagger 定义接口，自动生成文档
  • 工具描述：为每个工具提供详细的描述，包括用途、参数、返回值、示例
  • 版本文档：为每个版本维护独立的文档，记录变更历史
  • 最佳实践：提供最佳实践文档，指导开发者正确使用工具

接口契约：

  • Schema 定义：使用 JSON Schema 定义接口的输入输出格式
  • 类型检查：在运行时进行类型检查，确保参数类型正确
  • 契约测试：自动生成契约测试，验证服务是否符合接口契约
  • 版本兼容性：通过契约定义明确版本兼容性规则

LLM 工具理解：

  • 工具描述优化：优化工具描述，确保 LLM 能正确理解工具用途
  • 示例生成：为每个工具生成使用示例，帮助 LLM 学习正确用法
  • 语义搜索：支持 LLM 通过语义搜索找到合适的工具
  • 工具推荐：基于上下文推荐合适的工具，提高工具使用准确性

3.6 架构演进与迁移（Architecture Evolution & Migration）

遗留系统整合：

适配器模式：

  • API 适配器：为非 MCP 服务提供适配器，将其包装为 MCP 服务
  • 协议转换：支持不同协议（REST、gRPC、GraphQL）到 MCP 协议的转换
  • 数据格式转换：支持不同数据格式之间的转换，确保兼容性

渐进式集成：

  • 并行运行：支持新旧系统并行运行，逐步迁移功能
  • 数据同步：在迁移期间保持新旧系统的数据同步
  • 流量切换：支持逐步切换流量到新系统，降低迁移风险
  • 回滚机制：当新系统出现问题时，快速回滚到旧系统

迁移策略：

评估阶段：

  • 系统评估：评估现有系统的架构、功能、性能，识别迁移难点
  • 依赖分析：分析系统间的依赖关系，制定迁移顺序
  • 风险评估：评估迁移风险，制定风险应对措施

规划阶段：

  • 迁移计划：制定详细的迁移计划，包括时间表、里程碑、责任人
  • 测试计划：制定测试计划，确保迁移后系统功能正确
  • 回滚计划：制定回滚计划，确保迁移失败时可以快速回滚

执行阶段：

  • 分阶段迁移：采用分阶段迁移策略，先迁移非关键功能，再迁移关键功能
  • 持续监控：在迁移过程中持续监控系统状态，及时发现问题
  • 用户沟通：及时与用户沟通迁移进度和影响，管理用户期望

技术债务管理：

债务识别：

  • 代码质量分析：分析 LLM 生成代码的质量，识别技术债务
  • 架构债务：识别架构层面的技术债务，如设计不合理、性能瓶颈
  • 依赖债务：识别依赖层面的技术债务，如过时的依赖、安全漏洞

债务管理：

  • 债务登记：建立技术债务登记系统，记录所有技术债务
  • 优先级排序：根据影响和成本对技术债务进行优先级排序
  • 定期偿还：制定技术债务偿还计划，定期偿还高优先级债务
  • 预防机制：建立预防机制，避免产生新的技术债务

4. 场景演练

典型案例：药物研发 Agent

本节通过药物研发 Agent 的完整案例，展示本架构如何支持复杂的多工具协同场景。我们将演示架构的各个组件如何在实际业务中协同工作。

业务背景

药物研发是一个典型的复杂领域问题，涉及多个阶段和大量工具：

药物研发流程：

  1. 靶点识别：分析疾病相关基因、蛋白质结构
  2. 化合物设计：基于靶点设计候选化合物
  3. 分子模拟：计算化合物的物理化学性质
  4. ADMET 预测：预测吸收、分布、代谢、排泄、毒性
  5. 合成路线规划：设计化合物的合成路径
  6. 实验验证：进行体外和体内实验
  7. 数据分析：处理实验数据，评估效果

工具生态示例：

数据查询工具：

  • query_pubmed: 查询 PubMed 文献数据库
  • query_chembl: 查询 ChEMBL 化合物数据库
  • query_pdb: 查询蛋白质结构数据库
  • query_drugbank: 查询药物信息数据库

计算工具：

  • molecular_docking: 分子对接计算
  • molecular_dynamics: 分子动力学模拟
  • quantum_chemistry: 量子化学计算
  • admet_prediction: ADMET 性质预测
  • toxicity_prediction: 毒性预测

分析工具：

  • structure_analysis: 分子结构分析
  • similarity_search: 相似性搜索
  • clustering: 化合物聚类分析
  • visualization: 结果可视化

实验工具：

  • synthesis_planning: 合成路线规划
  • reaction_prediction: 反应预测
  • experiment_design: 实验设计
  • data_analysis: 实验数据分析

架构映射：全链路执行视图

场景：设计新型抗肿瘤药物

以下展示架构的各个组件如何在这个场景中协同工作：

  1. LLM 生成控制代码：

  2. 沙箱执行控制代码（对应：生命周期管理）：

    • 从预热池中获取沙箱实例，启动时间 < 200ms
    • 沙箱保持会话状态，支持多轮代码执行
    • 控制代码动态调用多个 MCP 服务
    • 支持并行调用、条件分支、循环等复杂逻辑

  3. 服务发现与调用（对应：MCP 服务发现与可扩展性）：

    • 沙箱通过服务注册中心发现可用的 MCP 服务
    • 负载均衡器选择最优的服务实例
    • 服务健康检查确保调用的服务可用

  4. MCP 服务提供功能代码（对应：控制代码与功能代码的职责边界）：

    • molecular_docking 服务：运行在 GPU 集群，需要 CUDA 环境
    • admet_prediction 服务：运行在机器学习推理集群
    • query_pubmed 服务：连接外部 API，需要网络访问
    • data_analysis 服务：需要大数据处理环境

  5. 状态与数据管理（对应：状态管理与数据流、数据存储架构）：

    • 工具间通过共享存储传递大文件（如化合物数据库）
    • 中间结果存储在对象存储中，通过引用传递
    • 会话状态存储在 Redis 中，支持快速恢复

  6. 错误处理（对应：错误处理与容错机制）：

    • 当 molecular_docking 超时，自动重试或降级到简化算法
    • 部分工具失败时，LLM 重新规划 workflow
    • 所有错误信息回传给 LLM，支持自愈循环

  7. 监控与追踪（对应：监控与可观测性）：

    • 全链路追踪：从用户请求到每个工具调用都有唯一 trace_id
    • 性能监控：记录每个工具的执行时间和资源使用
    • 业务指标：任务完成率、工具调用成功率

架构验证

通过这个案例，我们验证了架构设计的完备性：

  • ✅ 灵活的工具组合：LLM 可以根据问题动态生成工具调用序列
  • ✅ 并行执行：支持同时调用多个工具（分子对接、ADMET预测、毒性预测），提高效率
  • ✅ 环境隔离：不同工具运行在各自适合的环境中（GPU集群、ML推理集群、大数据环境）
  • ✅ 快速迭代：沙箱快速启动（<200ms），支持快速试错和迭代
  • ✅ 工具复用：功能代码一次编写，多次复用
  • ✅ 独立更新：工具可以独立更新，不影响其他组件
  • ✅ 健壮性：错误处理和自愈机制确保系统可靠运行
  • ✅ 可观测性：全链路追踪和监控支持问题定位和优化

5. 沙箱实现方案调研 ( draft )

5.1 AWS Firecracker

AWS 开源的虚拟机管理器，专为无服务器计算设计。microVM 可在 125ms 内启动，基于 KVM 提供硬件级安全隔离。

  • 官方网站：https://firecracker-microvm.github.io/
  • GitHub：https://github.com/firecracker-microvm/firecracker

5.2 E2B

开源运行时环境，专为 AI 生成代码执行设计。启动时间 <200ms，基于 Firecracker，支持多语言和长时间会话。

  • 官方网站：https://e2b.dev/
  • GitHub：https://github.com/e2b-dev/e2b

5.3 阿里云函数计算

阿里云的 FaaSNet 系统，用于加速无服务器容器部署。能在 8.3 秒内在 1000 台虚拟机上部署 2500 个函数容器。

  • 官方网站：https://www.alibabacloud.com/product/function-compute

5.4 字节跳动 SandboxFusion

字节跳动开源的安全沙箱，支持 Python、C++、Go、Java 等多种编程语言，适用于代码评测和在线编程场景。

  • GitHub：https://github.com/bytedance/SandboxFusion
