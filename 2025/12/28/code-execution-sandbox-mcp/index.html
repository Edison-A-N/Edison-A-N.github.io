1. 引言：从函数调用到代码执行的演进

随着 LLM Agent 能力的不断提升，Code Agents（代码执行 Agent）正成为 Agent 架构的重要演进方向。与传统的基于 JSON 的工具调用不同，Code Agents 让 LLM 生成可执行的代码，从而获得更强的表达能力、组合性和灵活性。本文提出了一种结合代码执行沙箱和 MCP（Model Context Protocol）服务的架构设计，解决复杂工具编排场景中的安全、效率和可扩展性挑战。

1.1 核心挑战

当 Agent 解决领域问题涉及大量的工具的拼接组装，且拼接组装的 workflow 是由 LLM 理解和规划的时，我们需要有工具的管理能力和复杂的工具调用组织能力。

关键挑战：

  • 工具数量庞大：领域问题往往需要调用数十甚至上百个不同的工具
  • 工具组合复杂：工具之间的调用顺序、依赖关系、并行执行等需要动态规划
  • Workflow 动态生成：LLM 需要根据问题理解动态生成执行计划
  • 工具管理复杂：不同工具有不同的执行环境、依赖、版本等

1.2 传统 Function Calling 的局限性

当 LLM 通过 Function Calling 逐个调用工具时，存在明显局限：

串行执行效率低（多步推理缺乏原子性）：

上下文窗口浪费：

  • 每次工具调用都需要将完整上下文传递给 LLM
  • Token 消耗巨大，特别是处理大量数据时
  • 无法有效利用并行执行能力

错误恢复困难：

  • 单个工具失败后，需要 LLM 重新理解整个流程
  • 难以局部修复，往往需要重新规划整个 workflow

安全性与灵活性矛盾：

  • 要么牺牲安全性（允许代码直接访问资源）
  • 要么牺牲灵活性（限制为简单的函数调用）

核心问题：传统 Function Calling 模式将多步推理拆解为多次往返，导致延迟累积、上下文爆炸、无法并行执行。我们需要一种能够一次性表达复杂逻辑的机制。

1.3 破局：为什么 LLM 应该写代码而不是发 JSON？

核心洞察：让 LLM 生成可执行的代码，而非逐条发送 JSON 函数调用。

研究支持：多项研究（如 Executable Code Actions Elicit Better LLM Agents）表明，使用代码而非 JSON 来表达工具调用具有显著优势。正如 Hugging Face 的 smolagents 文章所指出的：”我们专门设计了代码语言，使其成为表达计算机执行操作的最佳方式。如果 JSON 片段是更好的表达方式，JSON 就会成为顶级编程语言，编程就会成为人间地狱。“

关键优势（基于 Code Agents 研究）：

  • ✅ 组合性（Composability）：可以嵌套代码操作，定义函数复用逻辑，而 JSON 难以表达这种结构
  • ✅ 对象管理（Object Management）：代码可以自然地存储和管理操作结果（如 image = generate_image()），而 JSON 难以处理对象引用
  • ✅ 通用性（Generality）：代码语言设计用于表达任何计算机操作，表达能力远超 JSON
  • ✅ 训练数据优势：LLM 的训练数据中包含大量高质量代码，模型已经熟悉代码表达方式
  • ✅ 一次生成，批量执行：LLM 生成完整代码，减少交互次数
  • ✅ 降低 Token 消耗：数据在沙箱内处理，无需传回 LLM
  • ✅ 错误自愈：代码执行错误可直接反馈给 LLM 修复

1.3.1 重要权衡：调试复杂度与可验证性

架构设计如何缓解这些问题：

  1. 结构化错误信息：

    • 代码执行错误时，提供结构化的错误上下文（堆栈、变量状态、执行历史）
    • 将复杂的代码错误转换为 LLM 可理解的错误描述

  2. 分层错误处理：

    • 技术错误（语法错误、类型错误）：LLM 可以自动修复
    • 业务错误（数据不一致、逻辑错误）：需要更多上下文，可能需要人工介入

  3. 增量验证：

    • 支持代码的增量执行和检查点（Checkpoint）
    • 在关键步骤后验证中间结果，及早发现问题

  4. 错误恢复机制：

    • 当代码执行失败时，提供完整的错误上下文给 LLM
    • LLM 基于错误信息重新生成代码，实现自愈循环（详见 4.4 节）

何时代码执行仍然值得：

尽管存在调试复杂度问题，代码执行在以下场景仍然具有明显优势：

  • ✅ 复杂编排逻辑：需要并行执行、条件分支、循环等复杂控制流
  • ✅ 大量工具调用：需要调用数十甚至上百个工具，JSON 调用效率低下
  • ✅ 数据处理密集：需要在工具调用之间进行复杂的数据处理
  • ✅ 动态规划：需要根据中间结果动态调整执行计划

何时 JSON 调用更合适：

  • ✅ 简单工具调用：只需要调用1-3个工具，无需复杂编排
  • ✅ 高可靠性要求：需要每个步骤都有清晰的错误边界
  • ✅ 调试优先：需要快速定位和修复问题
  • ✅ 工具调用独立：工具之间没有复杂的数据依赖关系

结论：

代码执行不是银弹，而是一种权衡。本文提出的架构通过以下方式缓解代码执行的调试复杂度问题：

  1. 沙箱隔离：将代码执行限制在安全环境中，防止错误扩散
  2. 结构化错误处理：提供完整的错误上下文，帮助 LLM 理解问题
  3. 自愈机制：LLM 基于错误信息自动修复代码
  4. 职责分离：将复杂逻辑（代码）与能力实现（MCP Tools）分离，降低调试复杂度

这种架构设计使得代码执行的优势（表达能力、组合性）能够充分发挥，同时通过架构层面的设计缓解其劣势（调试复杂度、可验证性）。

2. 核心范式：编排逻辑与能力实现的分离

2.1 为什么需要分离？

问题根源：LLM 生成的脚本具有不可预测性和不可信性。

  • 不可预测性：LLM 可能生成错误的代码、死循环、资源耗尽等
  • 不可信性：即使代码逻辑正确，也不能信任它访问敏感资源

传统方案的局限性：

  1. 直接执行生成脚本（危险）：

  2. 限制为函数调用（不灵活）：

    • 无法支持复杂的编排逻辑（并行执行、条件分支、循环）
    • 每次调用都需要 LLM 参与，效率低下

核心洞察：我们需要一种架构，既能利用代码的表达能力（支持复杂控制流），又能保证安全性（防止直接访问敏感资源）。这需要将计算逻辑与副作用操作分离。

2.2 分离的设计原则

核心思想：LLM 充当程序员（生成编排脚本），MCP 充当标准库（提供原生工具）。

计算与 IO 的分离：

  • 生成脚本的职责（由 LLM 动态生成，负责计算逻辑）：

  • ✅ 动态组织工具的执行流程（if-else、循环、并行执行）

    • ✅ 基本数据处理（变量赋值、列表操作、字符串处理、数据转换）

  • ✅ 流程控制（条件判断、循环、异常处理）

    • ✅ 胶水逻辑（组合多个工具调用的结果）

  • ❌ 禁止：直接访问敏感资源（数据库、文件系统、网络）

  • ❌ 禁止：调用未注册的工具

  • 原生工具（MCP Tools）的职责（预定义的工具实现，负责副作用操作）：

  • ✅ 提供经过安全审计的功能实现

  • ✅ 处理复杂的执行环境需求（GPU、数据库、外部 API）

  • ✅ 封装敏感操作（数据库访问、文件操作、网络请求）

  • ✅ 一次编写多次执行，支持独立更新发布

  • ✅ MCP 映射：原生工具就是 MCP 协议中定义的 Tools，通过 MCP Server 暴露给生成脚本调用

为什么选择 MCP 而不是普通 OpenAPI？

  1. 标准化接口：MCP 提供统一的工具调用协议，Agent 不需要为每个工具重新学习 Schema
  2. 预植入能力：MCP Client 可以预植入沙箱中，生成脚本直接调用，无需动态发现
  3. 代码执行集成：MCP 协议原生支持代码执行能力，可以安全地执行 LLM 生成的脚本
  4. 工具发现机制：MCP 提供标准化的工具发现和注册机制，支持动态工具管理
  5. 协议成熟度：MCP 是 Anthropic 提出的标准协议，有完整的规范和实现参考

2.4 分离带来的价值

  1. 安全性：生成脚本无法直接访问敏感资源，所有敏感操作必须通过预定义的 MCP 原生工具
  2. 灵活性：支持复杂的编排逻辑，LLM 可以生成任意控制流
  3. 可扩展性：原生工具独立部署和更新，不影响生成脚本
  4. 效率：支持并行执行、批量调用，减少 LLM 交互次数
  5. 可维护性：原生工具集中管理，便于安全审计和版本控制

原生工具服务化的原因：

  • 安全性：防止 LLM 生成的脚本直接访问敏感资源
  • 一次编写多次执行：原生工具可复用，避免重复编写
  • 独立更新发布：原生工具可独立于生成脚本进行更新和发布
  • 复杂执行环境需求：原生工具有复杂的执行环境需求，不适合在轻量级沙箱中部署
  • 沙箱定位：沙箱更适合轻量级环境，支持快速唤起和脚本执行，降低延迟
  • MCP 标准化：原生工具通过 MCP 协议标准化暴露，生成脚本作为 MCP Client 调用这些工具

3. 系统架构设计

3.1 架构概览

架构通过职责分离实现安全、高效的工具编排：LLM 生成编排脚本在轻量级沙箱中执行，通过 MCP 协议调用原生工具（MCP Tools），实现复杂工具的组合编排。生成脚本是 MCP Tools 的消费者，原生工具通过 MCP Server 暴露为标准的 MCP Tools。

3.3.1 LLM 控制层（Brain）

职责：生成编排脚本，实现 Think -> Generate Code -> Execute -> Observe 循环。

核心机制：

  • 脚本生成：根据任务需求生成编排脚本，组织工具调用序列
  • 错误处理：当脚本执行失败时，将错误信息反馈给 LLM，触发自愈循环

3.3.2 代码执行沙箱（Runtime）

职责：提供安全隔离的执行环境，执行 LLM 生成的编排脚本。

核心要求：

  • 安全隔离：防止代码逃逸和资源泄露
  • 快速启动：支持快速创建和销毁，降低延迟
  • 资源限制：限制 CPU、内存、网络使用，防止资源耗尽

技术选型决策：

沙箱技术选择需要平衡安全性、性能、成本和运维复杂度四个维度。不同场景下的权衡因素如下：

1. 技术方案对比

维度 Docker 容器 MicroVM (Firecracker/gVisor) 进程沙箱 (seccomp/namespaces)
安全隔离 中等（共享内核） 高（独立内核/用户态内核） 低（仅系统调用过滤）
启动速度 快（~1秒） 中等（~2-5秒） 极快（<100ms）
资源开销 低（共享内核） 中等（独立内核） 极低（无额外开销）
适用场景 一般安全要求，高并发 高安全要求，多租户 低安全要求，极致性能

2. 选型决策树

3. 关键权衡因素

  • 安全隔离 vs 启动速度：MicroVM 提供更强的隔离，但启动时间更长；Docker 启动快但隔离较弱
  • 资源开销 vs 并发能力：MicroVM 资源开销大，适合高安全场景；Docker 资源开销小，适合高并发场景
  • 运维复杂度 vs 安全控制：MicroVM 需要更多运维工作，但提供更细粒度的安全控制

4. 实践方案对比

E2B 方案（smolagents 采用）：

  • 技术栈：基于 Firecracker MicroVM，提供强隔离
  • 优势：专为代码执行设计，安全隔离强，支持资源限制和网络控制
  • 适用场景：需要执行不可信代码的场景，多租户 SaaS 应用
  • 局限性：启动延迟相对较高，资源开销较大

注：Anthropic 提供了关于 MCP 协议在代码执行场景中的应用说明（参考 MCP 协议规范），但这主要是协议层面的设计思路，而非具体的沙箱实现方案。实际部署时，需要自行选择并实现沙箱技术（如 E2B、Docker、MicroVM 等）。

5. 选型建议

  • 企业内网场景：优先选择 Docker 容器，平衡安全性和性能
  • SaaS 多租户场景：优先选择 MicroVM（Firecracker/gVisor），确保强隔离
  • 高性能场景：考虑进程沙箱 + 严格白名单，牺牲部分隔离换取性能
  • 快速原型：使用 E2B 等托管服务，快速验证架构可行性

3.3.3 MCP 服务层（Connector）

职责：提供原生工具（MCP Tools）服务，封装工具实现。

MCP 的独特价值：

  • 标准化协议：统一的工具调用接口，降低集成复杂度（参考 MCP 协议规范）
  • 服务发现：支持工具注册和发现，动态更新工具列表
  • 负载均衡：支持多实例部署，实现高可用和扩展性
  • 代码执行集成：MCP 协议原生支持代码执行能力，可以安全地执行 LLM 生成的脚本

MCP 映射关系：

  • 原生工具 = MCP Tools：原生工具通过 MCP Server 暴露为标准的 MCP Tools
  • 生成脚本 = MCP Client：生成脚本在沙箱中作为 MCP Client 调用 MCP Tools
  • 职责分离：生成脚本负责编排逻辑，原生工具负责具体能力实现

MCP Tunneling 实现机制：

  1. 沙箱内注入 MCP Client Stub：

  2. 传输层选择：

    • STDIO：通过标准输入输出通信，适合单进程场景
    • Socket：通过 Unix Domain Socket 或 TCP Socket 通信，适合多进程/分布式场景
    • HTTP/WebSocket：通过 HTTP 或 WebSocket 通信，适合跨网络场景

  3. 权限控制：

    • 工具白名单：沙箱内只能调用注册过的工具，拒绝未授权调用
    • 参数验证：在 MCP Host 端验证参数格式和权限
    • 调用日志：记录所有工具调用，便于审计和调试

关键设计决策：

  • 为什么不在沙箱内直接运行 MCP Server？

    • 沙箱环境受限，无法访问外部资源（数据库、文件系统、网络）
    • MCP Server 需要访问敏感资源，必须在宿主机或 Sidecar 进程中运行

  • 为什么需要 Bridge？

    • 沙箱与宿主机之间的通信需要协议转换（Python 函数调用 → JSON-RPC）
    • Bridge 提供权限控制和错误处理能力

  • 网络隔离策略：

    • 沙箱默认无网络访问权限
    • 仅允许通过 Bridge 访问 MCP Server 端点
    • 禁止直接访问其他外部服务

3.3.5 数据存储层

职责：提供数据持久化和工具注册能力。

核心机制：

  • 引用传递：大数据采用引用传递，避免在生成脚本中传输大文件
  • 工具注册：维护 MCP Tools 元数据和语义索引，支持工具发现
  • 状态管理：支持会话状态持久化，实现多轮对话

3.4 关键特性

6. 生产级挑战：安全与性能

6.1 安全设计（架构的一等公民）

6.1.1 多层防护体系

不依赖单一安全机制，建立多层防护：

  1. 静态分析层：

    • AST 分析检测危险操作
    • 白名单机制限制可用模块
    • 局限性：动态语言难以完全检测，需配合其他机制

  2. 运行时监控层：

    • 监控代码执行行为
    • 检测异常模式（资源耗尽、无限循环）
    • 记录所有敏感操作

  3. 沙箱隔离层：

    • 容器/MicroVM 隔离
    • 资源限制（CPU、内存、网络）
    • 防止代码逃逸

  4. 工具白名单层：

    • Sidecar 进程校验每个工具调用
    • 拒绝未授权的工具调用
    • 记录所有工具调用日志

6.1.2 安全威胁与缓解措施

威胁 1：容器逃逸

  • 风险：MicroVM 隔离失效
  • 缓解：使用成熟的容器技术（Firecracker、gVisor），定期安全审计

威胁 2：权限提升

  • 风险：通过工具组合实现越权操作
  • 缓解：最小权限原则，调用链分析，检测异常组合模式

威胁 3：数据泄露

  • 风险：通过合法工具组合泄露敏感信息
  • 缓解：数据脱敏，访问控制，审计日志

威胁 4：拒绝服务

  • 风险：资源耗尽导致系统不可用
  • 缓解：资源配额，请求限流，超时控制

6.1.3 Prompt 防护

问题：依赖”多模型共识”防范提示词注入存在成本与延迟问题。

务实方案：

  1. 输入验证：严格的输入格式验证和清理
  2. 输出过滤：对 LLM 生成的脚本进行后处理
  3. 沙箱隔离：即使代码被注入，沙箱环境限制其危害
  4. 行为监控：监控代码执行行为，而非仅依赖输入检测

6.3 可靠性机制

6.3.1 错误处理策略

分层错误处理：

  1. 技术错误（可自愈）：

    • 语法错误、类型错误 → LLM 自动修复
    • 工具调用超时 → 重试机制

  2. 业务错误（需人工介入）：

    • 数据库死锁 → 需要领域知识
    • 数据不一致 → 需要业务上下文

改进方向：

  • 增强错误上下文：提供更多业务信息和系统状态
  • 错误分类：区分可自愈错误和需要人工处理的错误
  • 人工介入机制：识别无法自动修复的错误，及时上报

6.2.2 重试策略

分离策略：

  • 短期重试：基于规则的快速重试（3 次）
  • 长期优化：基于历史数据的策略优化（跨任务学习）

明确学习目标：

  • 定义清晰的优化指标
  • 建立评估机制验证学习效果
  • 渐进式学习：初期使用保守策略，随着数据积累逐步优化

6.3.3 分布式事务

挑战：在 LLM 生成的动态 workflow 中实现补偿操作几乎不可能。

务实方案：

  1. 最终一致性：接受短期不一致，通过后台任务最终一致
  2. 幂等性设计：所有工具操作设计为幂等，支持安全重试
  3. 业务补偿：在业务层面设计补偿流程，而非技术层面

6.4 可调试性（Debuggability）

核心挑战：当 Agent 执行失败时，如何快速定位问题？失败可能来自多个层面：

  • LLM 生成的代码语法错误
  • MCP 工具调用失败
  • 沙箱资源耗尽（OOM）
  • 网络超时
  • 业务逻辑错误

6.4.1 错误分类与诊断

错误分类体系：

  1. 代码层面错误（可自愈）：

    • 语法错误（SyntaxError）：LLM 可以自动修复
    • 类型错误（TypeError）：LLM 可以自动修复
    • 导入错误（ImportError）：依赖管理问题

  2. 执行环境错误（需系统干预）：

    • 内存溢出（OOM）：需要调整资源配额
    • 超时（Timeout）：需要优化执行逻辑或增加超时时间
    • 网络错误：需要检查网络连接

  3. MCP 工具错误（需工具修复）：

    • 工具调用失败：工具实现问题
    • 参数验证失败：参数格式错误
    • 权限错误：权限配置问题

  4. 业务逻辑错误（需领域知识）：

    • 数据不一致：需要业务上下文理解
    • 逻辑错误：需要领域专家介入

7. 总结与展望

7.1 核心创新

本架构的核心创新在于生成脚本（编排逻辑）与原生工具（能力实现）的分离：

  • 生成脚本：由 LLM 动态生成，负责编排逻辑，在轻量级沙箱中执行，作为 MCP Client 调用原生工具
  • 原生工具（MCP Tools）：预定义的工具实现，封装敏感操作，通过 MCP Server 标准化暴露

这种分离解决了 LLM Agent 在复杂工具编排场景中的三个根本性挑战：

  1. 工具编排的复杂性：支持复杂的编排逻辑，提高执行效率
  2. 代码执行的安全性与灵活性矛盾：既保证安全性，又支持灵活性
  3. 工具管理的复杂性：原生工具独立部署和更新，支持大规模工具生态

与 MCP 协议的映射关系：

  • 生成脚本 = MCP Client（在沙箱中调用工具）
  • 原生工具 = MCP Tools（通过 MCP Server 暴露）
  • 这种映射关系符合 MCP 协议规范，降低了架构理解成本

7.2 关键价值

沙箱和 MCP 只是实现手段，核心价值在于职责分离带来的安全性、灵活性和可扩展性：

  • ✅ 安全性：生成脚本无法直接访问敏感资源，必须通过 MCP Tools
  • ✅ 灵活性：支持复杂的编排逻辑，LLM 可以生成任意控制流
  • ✅ 可扩展性：原生工具独立部署和更新，不影响生成脚本
  • ✅ 效率：支持并行执行、批量调用，减少 LLM 交互次数
  • ✅ 可维护性：原生工具集中管理，便于安全审计和版本控制
  • ✅ 标准化：通过 MCP 协议标准化工具接口，降低集成复杂度

7.3 未来方向

  1. 更智能的工具发现：基于任务语义自动推荐相关工具
  2. 更强大的错误恢复：结合领域知识库，提升自愈能力
  3. 更细粒度的安全控制：基于工具调用链的动态权限管理
  4. 更高效的资源利用：智能预热池管理，降低资源浪费

7.4 设计原则

  • 安全第一：宁可过度防护，不可存在安全漏洞
  • 务实可行：设计要考虑实际可行性和成本
  • 数据驱动：决策要基于数据，而非假设
  • 持续改进：系统设计要支持迭代优化
