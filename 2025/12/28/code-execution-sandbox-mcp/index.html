1. 引言：从函数调用到代码执行的演进

随着 LLM Agent 能力的不断提升，Code Agents（代码执行 Agent）正成为 Agent 架构的重要演进方向。与传统的基于 JSON 的工具调用不同，Code Agents 让 LLM 生成可执行的代码，从而获得更强的表达能力、组合性和灵活性。本文提出了一种结合代码执行沙箱和 MCP（Model Context Protocol）服务的架构设计，解决复杂工具编排场景中的安全、效率和可扩展性挑战。

1.1 核心挑战

当 Agent 解决领域问题涉及大量的工具的拼接组装，且拼接组装的 workflow 是由 LLM 理解和规划的时，我们需要有工具的管理能力和复杂的工具调用组织能力。

关键挑战：

  • 工具数量庞大：领域问题往往需要调用数十甚至上百个不同的工具
  • 工具组合复杂：工具之间的调用顺序、依赖关系、并行执行等需要动态规划
  • Workflow 动态生成：LLM 需要根据问题理解动态生成执行计划
  • 工具管理复杂：不同工具有不同的执行环境、依赖、版本等

1.2 传统 Function Calling 的局限性

当 LLM 通过 Function Calling 逐个调用工具时，存在明显局限：

串行执行效率低（多步推理缺乏原子性）：

上下文窗口浪费：

  • 每次工具调用都需要将完整上下文传递给 LLM
  • Token 消耗巨大，特别是处理大量数据时
  • 无法有效利用并行执行能力

错误恢复困难：

  • 单个工具失败后，需要 LLM 重新理解整个流程
  • 难以局部修复，往往需要重新规划整个 workflow

安全性与灵活性矛盾：

  • 要么牺牲安全性（允许代码直接访问资源）
  • 要么牺牲灵活性（限制为简单的函数调用）

核心问题：传统 Function Calling 模式将多步推理拆解为多次往返，导致延迟累积、上下文爆炸、无法并行执行。我们需要一种能够一次性表达复杂逻辑的机制。

1.3 破局：为什么 LLM 应该写代码而不是发 JSON？

核心洞察：让 LLM 生成可执行的代码，而非逐条发送 JSON 函数调用。

研究支持：多项研究（如 Executable Code Actions Elicit Better LLM Agents）表明，使用代码而非 JSON 来表达工具调用具有显著优势。正如 Hugging Face 的 smolagents 文章所指出的：”我们专门设计了代码语言，使其成为表达计算机执行操作的最佳方式。如果 JSON 片段是更好的表达方式，JSON 就会成为顶级编程语言，编程就会成为人间地狱。“

关键优势（基于 Code Agents 研究）：

  • ✅ 组合性（Composability）：可以嵌套代码操作，定义函数复用逻辑，而 JSON 难以表达这种结构
  • ✅ 对象管理（Object Management）：代码可以自然地存储和管理操作结果（如 image = generate_image()），而 JSON 难以处理对象引用
  • ✅ 通用性（Generality）：代码语言设计用于表达任何计算机操作，表达能力远超 JSON
  • ✅ 训练数据优势：LLM 的训练数据中包含大量高质量代码，模型已经熟悉代码表达方式
  • ✅ 一次生成，批量执行：LLM 生成完整代码，减少交互次数
  • ✅ 降低 Token 消耗：数据在沙箱内处理，无需传回 LLM
  • ✅ 错误自愈：代码执行错误可直接反馈给 LLM 修复

架构设计如何缓解这些问题：

  1. 结构化错误信息：

    • 代码执行错误时，提供结构化的错误上下文（堆栈、变量状态、执行历史）
    • 将复杂的代码错误转换为 LLM 可理解的错误描述

  2. 分层错误处理：

    • 技术错误（语法错误、类型错误）：LLM 可以自动修复
    • 业务错误（数据不一致、逻辑错误）：需要更多上下文，可能需要人工介入

  3. 增量验证：

    • 支持代码的增量执行和检查点（Checkpoint）
    • 在关键步骤后验证中间结果，及早发现问题

  4. 错误恢复机制：

    • 当代码执行失败时，提供完整的错误上下文给 LLM
    • LLM 基于错误信息重新生成代码，实现自愈循环（详见 4.4 节）

何时代码执行仍然值得：

尽管存在调试复杂度问题，代码执行在以下场景仍然具有明显优势：

  • ✅ 复杂编排逻辑：需要并行执行、条件分支、循环等复杂控制流
  • ✅ 大量工具调用：需要调用数十甚至上百个工具，JSON 调用效率低下
  • ✅ 数据处理密集：需要在工具调用之间进行复杂的数据处理
  • ✅ 动态规划：需要根据中间结果动态调整执行计划

何时 JSON 调用更合适：

  • ✅ 简单工具调用：只需要调用1-3个工具，无需复杂编排
  • ✅ 高可靠性要求：需要每个步骤都有清晰的错误边界
  • ✅ 调试优先：需要快速定位和修复问题
  • ✅ 工具调用独立：工具之间没有复杂的数据依赖关系

结论：

代码执行不是银弹，而是一种权衡。本文提出的架构通过以下方式缓解代码执行的调试复杂度问题：

  1. 沙箱隔离：将代码执行限制在安全环境中，防止错误扩散
  2. 结构化错误处理：提供完整的错误上下文，帮助 LLM 理解问题
  3. 自愈机制：LLM 基于错误信息自动修复代码
  4. 职责分离：将复杂逻辑（代码）与能力实现（MCP Tools）分离，降低调试复杂度

这种架构设计使得代码执行的优势（表达能力、组合性）能够充分发挥，同时通过架构层面的设计缓解其劣势（调试复杂度、可验证性）。

2. 核心范式：编排逻辑与能力实现的分离

2.1 为什么需要分离？

问题根源：LLM 生成的脚本具有不可预测性和不可信性。

  • 不可预测性：LLM 可能生成错误的代码、死循环、资源耗尽等
  • 不可信性：即使代码逻辑正确，也不能信任它访问敏感资源

传统方案的局限性：

  1. 直接执行生成脚本（危险）：

  2. 限制为函数调用（不灵活）：

    • 无法支持复杂的编排逻辑（并行执行、条件分支、循环）
    • 每次调用都需要 LLM 参与，效率低下

核心洞察：我们需要一种架构，既能利用代码的表达能力（支持复杂控制流），又能保证安全性（防止直接访问敏感资源）。这需要将计算逻辑与副作用操作分离。

2.2 分离的设计原则

核心思想：LLM 充当程序员（生成编排脚本），MCP 充当标准库（提供原生工具）。

重要澄清：瘦沙箱（Thin Sandbox）约束

本文提出的”控制流-功能流分离”并非基于”代码由谁生成”（LLM vs MCP），而是基于资源消耗的可预测性和执行环境的职责边界。为了避免”轻量级沙箱”定位迷失，我们明确界定：

  • 沙箱环境：仅安装 Python 标准库 + MCP Client SDK，不包含重型计算库（pandas、numpy、torch 等）
  • 约束机制：通过 AST 静态分析在执行前扫描代码，禁止复杂数学运算和非法库导入
  • 设计模式：采用 Facade Pattern，沙箱代码只能作为”胶水”，必须通过 MCP 调用外部能力

计算与 IO 的分离：

  • 生成脚本的职责（由 LLM 动态生成，负责编排逻辑，在瘦沙箱中执行）：

  • ✅ 动态组织工具的执行流程（if-else、有限循环、并行执行）

    • ✅ 基本数据处理（变量赋值、列表操作、字符串处理、简单数据转换）

  • ✅ 流程控制（条件判断、有限循环、异常处理）

    • ✅ 胶水逻辑（组合多个工具调用的结果）

  • ❌ 禁止：直接访问敏感资源（数据库、文件系统、网络）

  • ❌ 禁止：调用未注册的工具

  • ❌ 禁止：导入非白名单库（仅允许：json、datetime、mcp_sdk）

  • ❌ 禁止：无限循环（while True）和复杂数学运算（必须通过 MCP 调用）

  • 原生工具（MCP Tools）的职责（预定义的工具实现，负责副作用操作）：

  • ✅ 提供经过安全审计的功能实现

  • ✅ 处理复杂的执行环境需求（GPU、数据库、外部 API）

  • ✅ 封装敏感操作（数据库访问、文件操作、网络请求）

  • ✅ 一次编写多次执行，支持独立更新发布

  • ✅ MCP 映射：原生工具就是 MCP 协议中定义的 Tools，通过 MCP Server 暴露给生成脚本调用

为什么选择 MCP 而不是普通 OpenAPI？

  1. 标准化接口：MCP 提供统一的工具调用协议，Agent 不需要为每个工具重新学习 Schema
  2. 预植入能力：MCP Client 可以预植入沙箱中，生成脚本直接调用，无需动态发现
  3. 代码执行集成：MCP 协议原生支持代码执行能力，可以安全地执行 LLM 生成的脚本
  4. 工具发现机制：MCP 提供标准化的工具发现和注册机制，支持动态工具管理
  5. 协议成熟度：MCP 是 Anthropic 提出的标准协议，有完整的规范和实现参考

2.4 分离带来的价值

  1. 安全性：生成脚本无法直接访问敏感资源，所有敏感操作必须通过预定义的 MCP 原生工具
  2. 灵活性：支持复杂的编排逻辑，LLM 可以生成任意控制流
  3. 可扩展性：原生工具独立部署和更新，不影响生成脚本
  4. 效率：支持并行执行、批量调用，减少 LLM 交互次数
  5. 可维护性：原生工具集中管理，便于安全审计和版本控制

原生工具服务化的原因：

  • 安全性：防止 LLM 生成的脚本直接访问敏感资源
  • 一次编写多次执行：原生工具可复用，避免重复编写
  • 独立更新发布：原生工具可独立于生成脚本进行更新和发布
  • 复杂执行环境需求：原生工具有复杂的执行环境需求，不适合在轻量级沙箱中部署
  • 沙箱定位：沙箱更适合轻量级环境，支持快速唤起和脚本执行，降低延迟
  • MCP 标准化：原生工具通过 MCP 协议标准化暴露，生成脚本作为 MCP Client 调用这些工具

3. 系统架构设计

3.1 架构概览

架构通过职责分离实现安全、高效的工具编排：LLM 生成编排脚本在轻量级沙箱中执行，通过 MCP 协议调用原生工具（MCP Tools），实现复杂工具的组合编排。生成脚本是 MCP Tools 的消费者，原生工具通过 MCP Server 暴露为标准的 MCP Tools。

3.3.2 代码执行沙箱（Runtime）

职责：提供安全隔离的执行环境，执行 LLM 生成的编排脚本。

核心要求：

  • 安全隔离：防止代码逃逸和资源泄露
  • 快速启动：支持快速创建和销毁，降低延迟
  • 资源限制：限制 CPU、内存、网络使用，防止资源耗尽
  • 瘦沙箱约束：仅安装 Python 标准库 + MCP Client SDK，不包含重型计算库

关键设计决策：

  • 为什么禁止无限循环？：降低死循环风险，强制使用 parallel_execute 或有限迭代器
  • 为什么禁止导入重型库？：保持沙箱”轻量级”，复杂计算必须通过 MCP 调用外部服务
  • 为什么需要静态分析？：Rice 定理表明程序的非平凡属性不可判定，但我们可以通过 AST 分析检测明显的违规模式

5. 选型建议

  • 企业内网场景：优先选择 Docker 容器，平衡安全性和性能
  • SaaS 多租户场景：优先选择 MicroVM（Firecracker/gVisor），确保强隔离
  • 高性能场景：考虑进程沙箱 + 严格白名单，牺牲部分隔离换取性能
  • 快速原型：使用 E2B 等托管服务，快速验证架构可行性

3.3.3 MCP 服务层（Connector）

职责：提供原生工具（MCP Tools）服务，封装工具实现。

MCP 的独特价值：

  • 标准化协议：统一的工具调用接口，降低集成复杂度（参考 MCP 协议规范）
  • 服务发现：支持工具注册和发现，动态更新工具列表
  • 负载均衡：支持多实例部署，实现高可用和扩展性
  • 代码执行集成：MCP 协议原生支持代码执行能力，可以安全地执行 LLM 生成的脚本

MCP 映射关系：

  • 原生工具 = MCP Tools：原生工具通过 MCP Server 暴露为标准的 MCP Tools
  • 生成脚本 = MCP Client：生成脚本在沙箱中作为 MCP Client 调用 MCP Tools
  • 职责分离：生成脚本负责编排逻辑，原生工具负责具体能力实现

错误恢复机制：如果生成的脚本在沙箱运行报错，如何自动修复？

重要澄清：自愈机制的理论边界

本文提出的”自愈机制”并非追求完美的自动修复，而是确定性的故障处理。需要明确以下理论边界：

  1. 哥德尔不完备性：一个系统无法在自身内部定义自己的正确性。LLM 无法保证”修复代码的代码”是正确的，这需要元验证层，而元验证层又需要元-元验证层，导致无限回归。

  2. 停机问题：如果你试图让 LLM 检查代码是否会死循环（停机问题），这是不可能解决的。

  3. 生产实践：Google 的 Bard 在代码生成失败时，不是让 LLM”自愈”，而是回退到人工编写的安全模式。

改进方案：有限状态机（FSM）与确定性恢复：

关键设计原则：

  1. 不追求完美自愈：接受某些错误无法自动修复，及时上报人工处理
  2. 确定性恢复：每种错误类型有明确的处理策略，而非依赖 LLM 的”智能”
  3. Circuit Breaker：防止无限重试导致 Token 消耗失控
  4. 错误分类：区分可自愈错误（语法、超时、逻辑）和需要人工处理的错误（资源、业务）

局限性（理论边界）：

  • ❌ 无法解决停机问题：无法判断代码是否会死循环
  • ❌ 无法保证修复正确性：LLM 修复的代码可能引入新错误
  • ❌ 需要领域知识：某些错误（如数据库死锁）需要领域专家介入
  • ❌ 需要系统状态：某些错误（如资源竞争）需要理解系统状态
  • ❌ 需要业务上下文：某些错误（如数据不一致）需要业务上下文

生产实践建议：

  • ✅ 优先使用：语法错误、超时错误的自动修复
  • ⚠️ 谨慎使用：逻辑错误的自动修复（需要 RAG 支持）
  • ❌ 避免使用：资源错误和业务错误的自动修复（直接上报人工处理）

6.2 性能优化

6.2.1 沙箱预热池（基于排队论的动态伸缩）

问题：冷启动延迟影响用户体验。简单的”预热池”如果不考虑到达率（Arrival Rate）和服务率（Service Rate），会导致资源浪费或雪崩。

排队论基础：

假设请求到达服从泊松分布，服务时间服从指数分布，使用 M/M/c 排队模型：

  • **λ (Lambda)**：请求到达率（每秒请求数）
  • **μ (Mu)**：服务率（每个实例每秒处理的任务数）
  • c：服务实例数量（预热池大小）
  • **ρ (Rho)**：系统利用率 = λ / (c × μ)

Little’s Law：$L = \lambda W$

  • L：系统内平均任务数（包括等待和执行中的）
  • W：平均停留时间（等待时间 + 服务时间）

动态伸缩算法：

关键设计要点：

  1. 防止队列无限增长：当请求率 λ > c × μ 时，系统过载，必须动态扩容
  2. 僵尸实例检测：如果 LLM 生成死循环代码，实例永远不会释放，需要强制终止
  3. 健康检查：定期检查实例健康状态，自动替换故障实例
  4. 池倾斜度监控：检测热点问题（某些实例被频繁使用，其他实例闲置）

多维度监控指标：

  • 请求队列长度（Queue Length）
  • 实例利用率（Utilization = λ / (c × μ)）
  • 冷启动频率（Cold Start Rate）
  • 用户等待时间（P95/P99 Wait Time）
  • 僵尸实例数量（Zombie Instance Count）

6.2.2 成本模型与量化分析

问题：成本单位混淆，不同维度的量无法直接相加。多租户场景下的成本归属问题。

成本单位分析：

不同组件使用不同的成本单位，无法直接相加：

组件 成本单位 计费方式 示例
LLM tokens/1K 按输入输出计费 5000 tokens = $0.05
沙箱 vCPU-hours 按资源时间计费 2 vCPU × 0.1小时 = $0.02
存储 GB-months 按容量时间计费 100MB × 1小时 = $0.001
MCP 服务 API calls 按调用次数计费 15 calls = $0.03

问题：一个任务消耗 5000 tokens + 2 vCPU-minutes + 100MB-storage-hours + 15 API calls，这些不同维度的量无法直接相加。

解决方案：多目标成本优化：

成本归属机制（多租户场景）：

在多租户场景下，需要明确成本归属：

  1. 预热池复用：租户A的沙箱实例被租户B的请求复用

    • 解决方案：按实际使用时间分配成本，而非按实例创建者

  2. OOM导致实例销毁：沙箱执行代码时OOM导致实例被销毁

    • 解决方案：成本由触发OOM的任务承担，而非实例创建者

  3. MCP服务熔断：MCP服务因某个沙箱的恶意调用被熔断，影响其他租户

    • 解决方案：熔断损失由触发熔断的租户承担，其他租户的损失计入系统成本

成本优化的帕累托前沿：

成本优化是多目标优化问题：

  • 降低 Token 可能增加执行时间（需要更多次LLM调用）
  • 减少预热池可能增加冷启动延迟
  • 优化需要找到帕累托前沿（Pareto Frontier）

ROI分析：

6.2.3 上下文窗口管理

关键优化：

  • 数据在沙箱内处理，无需传回 LLM
  • LLM 只看摘要（Summary），而非原始数据
  • 这是企业级应用的关键，保护数据隐私同时降低 Token 消耗

6.3 可靠性机制

6.3.1 错误处理策略

分层错误处理：

  1. 技术错误（可自愈）：

    • 语法错误、类型错误 → LLM 自动修复
    • 工具调用超时 → 重试机制

  2. 业务错误（需人工介入）：

    • 数据库死锁 → 需要领域知识
    • 数据不一致 → 需要业务上下文

改进方向：

  • 增强错误上下文：提供更多业务信息和系统状态
  • 错误分类：区分可自愈错误和需要人工处理的错误
  • 人工介入机制：识别无法自动修复的错误，及时上报

6.2.2 重试策略

分离策略：

  • 短期重试：基于规则的快速重试（3 次）
  • 长期优化：基于历史数据的策略优化（跨任务学习）

明确学习目标：

  • 定义清晰的优化指标
  • 建立评估机制验证学习效果
  • 渐进式学习：初期使用保守策略，随着数据积累逐步优化

6.3.3 分布式事务

挑战：在 LLM 生成的动态 workflow 中实现补偿操作几乎不可能。

务实方案：

  1. 最终一致性：接受短期不一致，通过后台任务最终一致
  2. 幂等性设计：所有工具操作设计为幂等，支持安全重试
  3. 业务补偿：在业务层面设计补偿流程，而非技术层面

6.4 可调试性（Debuggability）

核心挑战：当 Agent 执行失败时，如何快速定位问题？失败可能来自多个层面：

  • LLM 生成的代码语法错误
  • MCP 工具调用失败
  • 沙箱资源耗尽（OOM）
  • 网络超时
  • 业务逻辑错误

6.4.1 错误分类与诊断

错误分类体系：

  1. 代码层面错误（可自愈）：

    • 语法错误（SyntaxError）：LLM 可以自动修复
    • 类型错误（TypeError）：LLM 可以自动修复
    • 导入错误（ImportError）：依赖管理问题

  2. 执行环境错误（需系统干预）：

    • 内存溢出（OOM）：需要调整资源配额
    • 超时（Timeout）：需要优化执行逻辑或增加超时时间
    • 网络错误：需要检查网络连接

  3. MCP 工具错误（需工具修复）：

    • 工具调用失败：工具实现问题
    • 参数验证失败：参数格式错误
    • 权限错误：权限配置问题

  4. 业务逻辑错误（需领域知识）：

    • 数据不一致：需要业务上下文理解
    • 逻辑错误：需要领域专家介入

6.4.2 可观测性实现：AST 注入与 Trace Context 透传

核心挑战：LLM 生成的代码是动态的，传统的 APM（Datadog、Prometheus）无法在代码部署前插桩。如果脚本里写了 mcp_call 卡住了，在分布式链路追踪里只能看到”Sandbox Execute”，看不到内部是哪一步卡住。

问题场景：

  • LLM 生成 def foo(): mcp_call("tool_a") 和 def bar(): mcp_call("tool_a")
  • 在 Jaeger 中，这两个调用都显示为 tool_a，无法区分业务语义
  • 如果 LLM 在同一任务中多次调用 tool_a，追踪系统无法知道”哪次是查询，哪次是更新”

关键权衡：

  • ✅ 优势：完整的执行轨迹，可以追踪动态生成代码的执行逻辑
  • ⚠️ 代价：
    • 改变代码的 Token 消耗（注入代码增加了代码量）
    • 引入额外的执行延迟（每次函数调用都有开销）
    • 需要 LLM 理解桩代码语义（否则生成的代码可能绕过监控）

生产实践：

引用的 OpenTelemetry 在 Python 中的自动 instrumentation 依赖 sys.settrace，但这在沙箱中通常被禁用（性能和安全原因）。因此，我们采用 AST 注入的方式，在执行前静态修改代码，而非运行时动态追踪。

可视化调试：

关键设计原则：

  1. 错误信息结构化：提供结构化的错误信息，而非纯文本
  2. 上下文完整：包含足够的上下文信息，帮助 LLM 理解问题

7. 总结与展望

7.1 核心创新

本架构的核心创新在于生成脚本（编排逻辑）与原生工具（能力实现）的分离：

  • 生成脚本：由 LLM 动态生成，负责编排逻辑，在轻量级沙箱中执行，作为 MCP Client 调用原生工具
  • 原生工具（MCP Tools）：预定义的工具实现，封装敏感操作，通过 MCP Server 标准化暴露

这种分离解决了 LLM Agent 在复杂工具编排场景中的三个根本性挑战：

  1. 工具编排的复杂性：支持复杂的编排逻辑，提高执行效率
  2. 代码执行的安全性与灵活性矛盾：既保证安全性，又支持灵活性
  3. 工具管理的复杂性：原生工具独立部署和更新，支持大规模工具生态

与 MCP 协议的映射关系：

  • 生成脚本 = MCP Client（在沙箱中调用工具）
  • 原生工具 = MCP Tools（通过 MCP Server 暴露）
  • 这种映射关系符合 MCP 协议规范，降低了架构理解成本

7.2 关键价值

沙箱和 MCP 只是实现手段，核心价值在于职责分离带来的安全性、灵活性和可扩展性：

  • ✅ 安全性：生成脚本无法直接访问敏感资源，必须通过 MCP Tools
  • ✅ 灵活性：支持复杂的编排逻辑，LLM 可以生成任意控制流
  • ✅ 可扩展性：原生工具独立部署和更新，不影响生成脚本
  • ✅ 效率：支持并行执行、批量调用，减少 LLM 交互次数
  • ✅ 可维护性：原生工具集中管理，便于安全审计和版本控制
  • ✅ 标准化：通过 MCP 协议标准化工具接口，降低集成复杂度

7.3 未来方向

  1. 更智能的工具发现：基于任务语义自动推荐相关工具
  2. 更强大的错误恢复：结合领域知识库，提升自愈能力
  3. 更细粒度的安全控制：基于工具调用链的动态权限管理
  4. 更高效的资源利用：智能预热池管理，降低资源浪费

7.4 设计原则

  • 安全第一：宁可过度防护，不可存在安全漏洞
  • 务实可行：设计要考虑实际可行性和成本
  • 数据驱动：决策要基于数据，而非假设
  • 持续改进：系统设计要支持迭代优化

8.3 附录C：生产事故案例研究

案例1：死循环导致预热池耗尽

场景描述：

  • LLM 生成了包含死循环的代码：while True: mcp_call("query_db", {...})
  • 沙箱执行该代码，实例永远不会释放
  • 预热池中的5个实例全部被占用，新请求无法获取实例，导致系统不可用

根本原因：

  • AST 静态分析未能检测到死循环（使用了 while True 但条件判断复杂）
  • 缺少执行超时机制
  • 缺少僵尸实例检测

解决方案：

  1. 强制禁止 while True 循环（AST 静态分析）
  2. 添加执行超时机制（5分钟强制终止）
  3. 实现僵尸实例检测和自动清理

经验教训：

  • 静态分析无法检测所有问题，需要运行时保护
  • 预热池需要健康检查和资源泄漏检测

案例2：MCP 服务熔断影响多租户

场景描述：

  • 租户A的沙箱代码恶意调用 MCP 服务，导致服务熔断
  • 租户B的正常请求也被拒绝，影响业务

根本原因：

  • MCP 服务缺少租户级别的限流和熔断
  • 成本归属不明确，无法追溯责任

解决方案：

  1. 实现租户级别的限流和熔断
  2. 成本归属机制：熔断损失由触发熔断的租户承担
  3. 监控和告警：及时发现异常调用模式

经验教训：

  • 多租户场景需要细粒度的资源隔离
  • 成本归属机制对于责任追溯至关重要

案例3：AST 注入导致 Token 消耗激增

场景描述：

  • 为可观测性注入的 AST 代码增加了代码量
  • LLM 生成的代码从 100 tokens 增加到 500 tokens
  • Token 消耗增加 5 倍，成本大幅上升

根本原因：

  • AST 注入的代码量未考虑 Token 成本
  • 注入策略过于激进，每个函数都注入 span

解决方案：

  1. 优化 AST 注入策略：只在关键函数注入
  2. 使用更轻量级的追踪机制（如采样）
  3. 监控 Token 消耗，设置阈值告警

经验教训：

  • 可观测性需要权衡成本和价值
  • 采样策略可以大幅降低追踪成本
